# 🚀 عرض تقديمي للتيم: نظام الـ Server Actions المطور

أهلاً يا شباب، ده ملخص للتعديلات اللي عملناها عشان نربط بين **React Hook Form** و **Zod** و **Next.js Server Actions** بطريقة احترافية تضمن الأمان وسهولة الاستخدام.

## 🛠️ ورا الكواليس (Behind the Scenes)

لما المستخدم بيدوس على زرار **إرسال**، إيه اللي بيحصل بالظبط؟

1.  **المرحلة الأولى (Client Validation):**
    - بنشغل `form.handleSubmit`.
    - بيتم التحقق من الحقول باستخدام **Zod Schema**.
    - لو فيه أخطاء (مثلاً: الاسم أقل من 3 حروف)، بنوقف العملية فوراً ونعرض الخطأ.

2.  **المرحلة الثانية (Server Action Trigger):**
    - لو المرحلة الأولى نجحت، بيتم استدعاء الـ `actionFn` (الـ Server Action).
    - بياخد البيانات من الفورم عن طريق `FormData`.

3.  **المرحلة الثالثة (Server Side Processing):**
    - السيرفر بيعمل **Re-validation** (إعادة فحص) للبيانات (عشان نضمن إن محدش دور حوالين الجافاسكريبت).
    - لو فيه أخطاء سيرفر (مثلاً: إيميل مستخدم من قبل)، السيرفر بيرجع الـ Errors دي.

4.  **المرحلة الرابعة (Syncing State):**
    - الهوك بتاعنا (`useServerForm`) بيستنى النتيجة.
    - لو فيه أخطاء سيرفر، بيعرضها فوراً تحت الحقول المناسبة (`form.setError`).
    - لو مفيش أخطاء، بيظهر **Success Toast** بالرسالة اللي السيرفر بعتها.

## ❓ أسئلة وأجوبة (Q&A)

### ليه عملنا الفاليشن مرتين؟

- **عشان الـ UX:** المستخدم مبيحبش يستنى رد السيرفر عشان يعرف إنه نسي يكتب حرف في حقل "الاسم".
- **عشان الـ Security:** السيرفر لازم يكون هو الحكم النهائي، ومينفعش نثق في أي حاجة جايه من المتصفح 100%.

### ليه بنبعت `onSubmit={() => {}}`؟

- دي الطريقة اللي بنربط بيها بين الـ `handleSubmit` بتاعة المكتبة وبين الـ `action` بتاع رياكت 19. هي بس بتضمن إن الفلديشن يشتغل قبل ما الأكشن يروح للسيرفر.

### إيه فايدة الهوك `useServerForm`؟

- بدل ما تكرر الكود بتاع عرض الـ Toast وتوزيع الأخطاء في كل صفحة، الهوك ده بيعمل كل ده أوتوماتيكياً لأي فورم في الموقع.
